# SkinCareAI 项目演示与交接视频讲稿

副标题：版本 2026-04-17｜建议时长 11 分钟左右｜用途：项目交接、答辩录屏、队长说明

说明：本讲稿按“时间段 -> 我说的话 -> 动作 -> 画面重点”组织，适合直接照稿录制。每个时间段内部会在 Word 中用不同底色区分“我说的话”和“动作”。

## 录制前准备

- 准备两个终端窗口：一个用于启动后端 `server`，一个用于启动前端 `web`。
- 浏览器提前关闭无关标签页，缩放保持在 `100%`，录屏窗口尽量固定为 `16:9`。
- 将 GitHub 仓库页、`README.md`、聊天页、模型页、知识页依次加入可快速切换的标签页。
- 准备一张皮肤示例图片，便于演示图片问诊、模型切换保草稿和视觉模型分析。
- 如需展示原生分享，建议另备移动设备；桌面演示可重点展示复制和 `Markdown` 导出。

## 00:00 - 00:40 | 开场与项目背景

说：大家好，这次演示的是 SkinCareAI 的 Web 版工作区。这个项目不是单纯做一个聊天页面，而是围绕皮肤健康咨询场景，把文本问诊、图片问诊、模型选择、结构化知识展示和专家会诊整合到同一套工作流里，目标是让它既能演示，也能实际交接和继续维护。

做：录屏从桌面开始，先打开项目对应的 GitHub 仓库首页或本地项目根目录，镜头稳定停留两到三秒，再切到仓库说明页。

重点：开场先讲项目定位和完整链路，不要一开始就直接点页面，否则观众只能看到界面，看不出项目解决了什么问题。

## 00:40 - 01:20 | 展示仓库与文档入口

说：当前交付不是只有代码，还配了完整的文档链路。根目录的 README 用来说明项目背景、运行方式、功能矩阵和页面结构；交接话术文件用于把仓库直接发给队长；演示讲稿则对应这次录屏流程。这样后续接手时，不需要先翻代码，就能先把项目跑起来、看明白、用起来。

做：打开 GitHub 仓库首页，随后点开根目录 `README.md`；鼠标缓慢划过 README 标题、目录和快速开始位置。

重点：这一段要突出“交接友好”和“资料完整”，说明本次整理不仅改了页面，也把接手成本降下来了。

## 01:20 - 02:10 | 讲 README 的结构

说：README 里先写了如何运行，再写功能总览、前端要求实现对照、页面说明和交付材料。这里最关键的是前端要求对照表，因为基础功能、扩展功能和专家增强都已经按模块归档，后续无论是验收、答辩还是继续开发，都可以直接按这份表定位。

做：在 `README.md` 中滚动到“快速开始”“功能总览”“前端要求实现对照”三个部分，各停留两到三秒。

重点：镜头只需要覆盖关键章节，不要大段慢滚，避免录屏信息密度过高。

## 02:10 - 03:00 | 现场启动后端

说：下面直接按 README 的方式启动项目。后端位于 `server` 目录，负责统一接模型目录、聊天接口、流式输出和视觉接口。这里也特别处理了端口冲突场景，如果 `8787` 被占用，说明本机已经有旧实例在运行，需要先清旧实例再继续。

做：切到终端，依次输入 `cd server`、`npm install`、`copy .env.example .env`、`npm run dev`，让镜头停在启动成功日志位置。若已有实例，则简单说明端口占用含义。

重点：终端要录清楚命令本身和最终监听端口，便于后续照着复现。

## 03:00 - 03:40 | 现场启动前端

说：前端位于 `web` 目录，开发端口固定为 `127.0.0.1:5173`，并启用了 `strictPort`。这样做的目的，是避免开发环境静默跳端口，导致页面和当前源码版本错位。这个细节对交接非常重要，因为只要打开错实例，页面就会看起来像没有改动。

做：切到第二个终端，依次输入 `cd web`、`npm install`、`npm run dev`，展示前端地址为 `http://127.0.0.1:5173`。

重点：这一段要点明“固定端口”和“实例核对”的意义，让后续接手少踩坑。

## 03:40 - 04:20 | 进入首页并说明整体布局

说：进入页面后，最先看到的是顶栏、工作区标签页和聊天主区。左上角是覆盖式抽屉入口，用来统一承接历史会话、帮助、免责声明和产品介绍；顶栏只保留品牌、主导航、主题切换、帮助和更多工具；聊天区则成为首屏最大主角。整体布局的原则就是高频交互前置，低频说明后置。

做：在浏览器打开 `http://127.0.0.1:5173/chat`，鼠标缓慢划过顶栏、标签页、聊天区和左上角抽屉按钮。

重点：这一段是页面结构总介绍，镜头不要急着点击，先让整体布局被看清楚。

## 04:20 - 05:10 | 演示文本问诊

说：先走最基础也是最核心的文本问诊链路。在输入框中输入一个皮肤症状问题，例如描述红斑、瘙痒或脱屑的持续时间和部位。发送后可以看到清晰的消息分层、生成中动画和流式回复。这里的目标是让用户第一眼就知道谁在说话、当前是否仍在生成，以及后续还能不能继续追问。

做：在输入区输入一条皮肤问题并点击发送，等待 AI 开始流式回复。回复生成期间把镜头停留在消息区，让“生成中”状态被完整录进去。

重点：文本问诊是第一优先级功能，务必录到完整发送、生成和回复收口的全过程。

## 05:10 - 06:10 | 演示图片问诊与草稿保留

说：接下来演示图片问诊。输入区支持上传、拖拽和粘贴图片，图片会先在发送前进入预览区。如果当前模型不支持视觉能力，系统会自动切换到推荐视觉模型并新建会话，同时把当前已经输入的文字和图片草稿一起带过去，这样就不会因为切模型导致内容丢失。

做：上传一张示例皮肤图片，在输入框补一句病情描述后发送。随后切换一次模型，展示草稿文字和图片仍然保留，再回到合适模型继续发送。

重点：这一段重点讲两个体验点：图片问诊本身，以及“切模型不丢草稿”的连续性。

## 06:10 - 07:00 | 演示侧边抽屉与会话管理

说：左上角抽屉负责承接整个会话管理链路。这里可以新建会话、搜索会话、按时间查看历史记录，也可以对单条会话做重命名、标签、归档和恢复。这个设计的核心是把低频管理入口集中到一个区域里，避免聊天主区被各种管理按钮打散。

做：点击左上角抽屉按钮，展示会话搜索、单条会话重命名、标签编辑，以及“进行中”和“已归档”两个视图。

重点：抽屉要从顶部滚到稍靠下的位置，确保帮助、免责声明和产品说明都能被完整看到。

## 07:00 - 08:10 | 演示模型工作台与全屏模式

说：接下来切到模型工作台。这里不是静态说明页，而是一个可操作的模型选择区：可以按分类浏览模型、查看当前状态和延迟、看最近检测时间，也可以手动触发连通性测试。模型页还支持全屏，这样在演示时可以一次看清更多模型，而不是在狭窄列表里来回滚动。

做：切换到“模型”标签页，点击模型工作台右上角全屏按钮，展示分类、模型列表、延迟和“连通性测试”按钮，再手动触发一次测试。

重点：这一段重点展示“模型选择不是黑箱”，而是可见、可比、可验证的。

## 08:10 - 08:50 | 演示“去咨询”与聊天全屏

说：模型页全屏里还补了一个很实用的小动作，就是右上角的“去咨询”。选定模型后，不需要先退出模型全屏、再切回聊天、再重新放大，而是一步直接跳回聊天页并自动进入聊天全屏。这样在现场演示时，模型选择和真正发问之间就不会断链。

做：在模型全屏里选定一个模型，点击“去咨询”，展示页面自动切回聊天标签并进入聊天全屏。随后退出聊天全屏，回到正常布局。

重点：镜头要完整录下点击按钮后的连续跳转，这个功能很适合当作用户体验优化点来讲。

## 08:50 - 09:30 | 演示导出、分享与语气设置

说：除了基本问诊，这一版也补了很多细节能力。顶栏支持会话复制、Markdown 导出和系统分享；单条 AI 回答右侧也能单独导出 Markdown。输入区内部还放了高级设置，可以切换回答语气，但默认折叠，不会把聊天主区挤压得很乱。

做：在已有 AI 回答旁点击单条导出图标，再演示顶栏的复制或导出按钮，最后展开输入区高级设置，展示语气选项组后收起。

重点：这一段强调“实用细节”和“不会打扰主流程”的平衡。

## 09:30 - 10:10 | 演示知识页

说：除了聊天，这个项目还保留了独立的知识页。知识页不是聊天页的附属说明，而是结构化内容展示区，用来承接皮肤知识卡片、分类导览、阅读提示和病例图演示。这样项目在答辩时不仅能展示生成式交互，也能展示静态知识组织和内容设计能力。

做：切换到“知识页”，展示顶部分类与第一屏知识卡片区，再向下轻微滚动，让病例图和说明区露出一部分。

重点：知识页主要录首屏和第一段滚动，不要整页慢速通滚。

## 10:10 - 10:50 | 演示专家会诊

说：在需要更强说服力的场景下，还可以打开专家会诊。这个功能会围绕当前问题并排比较多个模型的回答，再收口成一个更适合展示的会诊结果。它的价值不只是“多模型同时跑”，更重要的是把差异化观点可视化，让结果更适合答辩和演示。

做：通过顶栏“更多工具”打开专家会诊，展示会诊运行中和完成后的结果面板。

重点：会诊不用录太久，重点是让“运行中”和“完成态”都被看到。

## 10:50 - 11:30 | 总结项目亮点与交接顺序

说：最后总结一下，这个版本已经把基础功能、进阶功能和专家增强都整理进同一套 Web 工作区里，并且配好了 README、交接话术和录屏讲稿。后续接手顺序也很明确：先看 README，再看视频，再本地启动复现，最后按 `audit` 和 `memory` 继续维护。整个交付目标，是把项目从“只有代码”整理成“可运行、可讲解、可接手”的状态。

做：镜头回到 `README.md` 或项目根目录，停留在 `README.md`、`handoff_talk.md` 和 `SkinCareAI_demo_script.docx` 这几个文件上，最后结束录制。

重点：收尾一定回到交付材料，形成完整闭环。
